# テスト駆動開発ワークフロー

このスキルは、コード変更が **Red → Green → Refactor** のサイクルに従い、振る舞いを記述したテストが**先**に存在することを保証する。言語非依存。

## 発動タイミング

- 新しい機能・モジュール・関数の追加時
- バグ修正時（**まず再現テスト**を書いてから直す）
- 既存コードのリファクタリング時（テストが安全網になる）
- 公開API（CLI コマンド、HTTP エンドポイント、ライブラリ関数）を変更する時

## TDDの基本原則

### 1. Red: 失敗するテストを先に書く
- 期待する**振る舞い**を表すテストを最初に書く
- そのテストが**意図通り失敗すること**を確認する（実行するか、コンパイルが通って `assert` で落ちることを確認）
- 失敗の理由が「型が無い」「関数が無い」など想定外でないか確認する

### 2. Green: 最小限の実装でパスさせる
- テストをパスさせるためだけの最小コードを書く
- 過度な汎化、未来予測、便利機能の先取りはしない（YAGNI）
- ハードコードでもよい。次のテストで一般化を強制させる

### 3. Refactor: テストをグリーンに保ったまま整える
- 重複の除去、命名改善、構造化
- **テストを変えずに**コードを変える。テストを変える時は別コミットに分ける
- 各リファクタの前後でテストを実行する

## テストの三層

| 層 | 対象 | スピード | 隔離 |
|---|---|---|---|
| ユニット | 単一関数・クラスの純粋ロジック | ms 級 | 外部依存なし（あればテストダブルで隔離） |
| 統合 | 複数モジュール、I/O、DB、設定 | 〜数百ms | 一時環境（一時ディレクトリ、in-memory DB） |
| E2E | プロセス起動 / HTTP / CLI 実行 | 秒級 | 外部世界に近い |

**比率の目安**: ユニット 70% / 統合 20% / E2E 10%（テストピラミッド）。E2E は遅く壊れやすいので、各機能の代表シナリオに絞る。

## ワークフローステップ

1. **インターフェースを定義**: 関数シグネチャ・型・公開APIを宣言（実装は未定義のまま）
2. **テストケースを列挙**: 正常系1〜2、異常系（エラー条件）1〜2、境界値（空・最大・最小・null/None）
3. **最初のテストを書く**: 1ケースだけ。Arrange-Act-Assert で構造化
4. **実行 → Red を確認**: 失敗が想定通りか確認（型不在エラーは Red ではない）
5. **最小実装 → Green**: ハードコード可。テスト1個だけパスさせる
6. **次のテストを追加 → Red → Green** を繰り返す
7. **Refactor**: 重複除去、命名改善。テストはグリーンのまま
8. **カバレッジ確認**: 残ったギャップを次のテストで埋める

## テスト構造: Arrange-Act-Assert (AAA)

```
# 各テストはこの3区画に分かれる
def test_filter_by_project_returns_matching_tasks():
    # Arrange: 入力と前提を準備
    tasks = [task("a", project="web"), task("b", project="api")]

    # Act: 対象を1回だけ呼び出す
    result = filter_tasks(tasks, project="web")

    # Assert: 振る舞いを検証
    assert len(result) == 1
    assert result[0].project == "web"
```

- Act は**1回だけ**にする（複数の Act を1テストに混ぜない）
- Assert は**同じ概念**に絞る（複数 assert は OK だが、論点が分散しない範囲）

## 命名規約

テスト名は「**何を**検証しているか」を1文で表す:

```
# 良い: 具体的で説明的
test_add_task_with_empty_title_returns_error
test_filter_by_project_excludes_other_projects
test_status_change_to_done_archives_task

# 悪い: 曖昧
test_add
test_it_works
test_1
```

## 振る舞いをテストする、実装をテストしない

```
# ❌ 悪い: 内部状態（プライベートフィールド）に依存
assert service._cache.size == 1

# ✅ 良い: 公開APIの結果を検証
result = service.list_tasks(filter)
assert len(result) == 1
```

実装詳細をテストすると、リファクタのたびにテストが壊れて TDD の安全網が崩れる。**公開された振る舞い**だけを検証する。

## エッジケースの最低ライン

各機能で必ずカバーする:

- **空入力**: 空配列、空文字、null/None
- **境界値**: 最小、最大、ちょうど境界（off-by-one を狙う）
- **エラーパス**: 期待する例外/エラー型を返すか
- **冪等性が重要な操作**: 同じ呼び出しを2回しても壊れないか
- **入力非破壊性**: 関数が引数を変更しないこと（特に配列・オブジェクト）。デフォルトは「入力非破壊」を契約としてテストで固定する

## デフォルト動作の解釈ルール

「`flagX=false` がデフォルト」と仕様にある場合:
- `options` 引数を**完全に省略**した呼び出しでも、デフォルト動作が適用される（`f(x)` と `f(x, {})` と `f(x, { flagX: false })` は同じ）
- このルールを少なくとも 1 つのテストで明示的に検証すること（`f(x)` の形で呼んで挙動が同じか）
- 仕様が曖昧な場合はテスト名で前提を可視化（`returns_X_when_options_omitted`）し、仕様作成者と早期に齟齬を surface する

## テストの独立性

- テストは **任意の順序で実行できる** こと（順序依存テストは禁止）
- テスト間で **共有可変状態を持たない**（グローバル変数、共有DB レコード）
- セットアップ/ティアダウンで **毎回クリーン**な環境を作る（一時ディレクトリ、トランザクションロールバック）

## カバレッジの扱い

- カバレッジ**数値**は目的ではなく、ギャップ発見のレンズ
- 一般的な目安: **80% 以上**（複雑度の高い領域は **100% 必須**: 認可判定、状態遷移、永続化、入力サニタイズ）
- カバレッジが低い箇所は「テストを追加すべきか」「そのコードが本当に必要か」を問う
- **ブランチカバレッジ**（条件分岐の網羅）を行カバレッジより重視する

## 言語別の具体ツール（参考）

実コマンドは言語/フレームワークに依存。代表例:

| 言語/環境 | 実行 | カバレッジ | E2E |
|---|---|---|---|
| Rust | `cargo test` | `cargo tarpaulin` / `cargo llvm-cov` | `assert_cmd` + `predicates` |
| TypeScript / Node | `pnpm test` / `npm test` (vitest, jest) | `vitest --coverage` / `jest --coverage` | `playwright` / `supertest` |
| Python | `pytest` | `pytest --cov` (`coverage.py`) | `pytest` + `subprocess` / `httpx` |
| Go | `go test ./...` | `go test -cover` | `testscript` / `os/exec` |

**注意**: ツールは目的（Red-Green-Refactor を回す）の**手段**にすぎない。プロジェクトの慣習が優先。

## 避けるべきよくある間違い

- **Green までを跳ばす**: テストを書かずに実装、後で「テストを書く」と言って書かない
- **Red を確認せずに進む**: 「失敗するはず」と思い込んで実装。実は最初からパスしていた（テストが何も検証していない）
- **巨大なテスト**: 1テストで複数の振る舞いを検証 → どれが壊れたか分からない
- **モックでロジックを差し替える**: モックが本物の振る舞いと乖離し、テストはパスするのに本番で壊れる
- **テスト名が「正常系」「異常系」だけ**: 何を検証しているか読まないと分からない
- **失敗テストをコメントアウトしてマージ**: テストの意味が消滅する。直すか削除を選ぶ

## バグ修正時の TDD

1. バグを**再現するテスト**を書く（Red になることを確認）
2. テストをパスさせる最小修正を行う（Green）
3. 同じバグの**亜種**を考えて追加テストを書く（境界値、似た条件）
4. リグレッションテストとしてコミット

「先に再現テスト」を省くと、修正が間違っていても気付けず、同じバグが再発する。

**バグの実コードが提供されない / 仮想で議論する場合**:
- 「想定するバグ実装」を先に**明示**してから再現テストを書く（暗黙の仮実装を読者が推測する状態にしない）
- 例外メッセージのテキスト一致は契約に明記された場合のみアサート。それ以外は型のアサートに留める（実装詳細とのカップリングを避ける）
- 失敗テストのみのコミットは PR 内では separate commit として残してよいが、**マージ単位は test+fix を 1 PR にまとめる**（CI が常に green になる単位）

## チェックリスト

機能完了の判断:

- [ ] 公開APIの正常系テストが存在する
- [ ] 公開APIの異常系（エラーパス）テストが存在する
- [ ] 境界値（空、最大、最小）テストが存在する
- [ ] テストが**実装前に Red を確認**してから書かれた、または等価な確証がある
- [ ] 全テストが独立に実行できる（順序依存なし）
- [ ] 内部状態ではなく振る舞いをテストしている
- [ ] カバレッジが目標を満たす（または不足箇所を意識的に許容している）

---

**注意**: TDD の本質はテストを書くことではなく、**Red を見てから Green に進む規律**。テストを後付けする「テストファースト風」ではこの規律が失われる。失敗を1度確認してから実装に進むこと。
