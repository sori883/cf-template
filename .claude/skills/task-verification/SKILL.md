# タスク検証スキル

`todos` CLI で管理しているタスクの要件（`content` フィールド）と実装コードを突き合わせ、仕様適合性を検証する。

**前提**: タスクは `todos` CLI で管理されている。要件は `todos edit <ID> --content "..."` で記録済み。

## 発動タイミング

- 「タスクの実装が合っているか確認して」「仕様通りか検証して」と指示された場合
- `todos status <ID> done` の前のゲートチェック

## 検証フロー

### Step 1: タスク情報を取得

```bash
todos show <ID>
todos list -P <project> --format json
```

抽出する情報:
- **タスクタイトル**: 実装対象の概要
- **content**: 要件・設計メモ・テストケース定義
- **サブタスク**: 個別の実装ステップとステータス

### Step 2: 要件を構造化

`content` から以下を分離する:
- 機能要件（何を実装するか）
- 制約・バリデーション（上限値、入力制限等）
- テストケース定義（正常系・異常系・境界値）

### Step 3: 実装コードを特定

1. content にファイルパスがあればそれを使う（`Read` で開く）
2. なければキーワードでコードベースを検索
   - **タスクタイトル・要件文中の主要名詞/動詞**を抽出してクエリ化
   - `Grep` で `pattern="<キーワード>"` を実行（複数キーワードは `|` で OR）
   - **ヒット数が 30 件超 or 5 ディレクトリ以上に分散**した場合は `Glob` でパス絞り込み（`src/**/*.ts` 等）
   - 0 件ヒット時は (a) タスクタイトル全文を Grep に当てる、(b) `git log --grep="<keyword>" -i` でコミット履歴を辿る、(c) それでも見つからなければ `iterative-retrieval` スキルの 4 フェーズで広げて絞る
3. 関連するテストファイルも併せて特定（`*.test.*` / `*_test.*` / `tests/` ディレクトリ）

### Step 4: 要件ごとに突合

| 観点 | 確認内容 |
|------|---------|
| **存在性** | 要件に対応するコードが存在するか |
| **正確性** | 実装が仕様通りの動作をするか |
| **完全性** | 要件の全項目がカバーされているか |
| **整合性** | 制約が守られているか |
| **副作用** | 仕様にない変更や既存機能への影響がないか |

### Step 5: 検証レポートを出力

## レポートフォーマット

```markdown
# タスク検証レポート: {タスクタイトル}

対象: <ID> ({プロジェクト名})

## サマリ

| 項目 | 結果 |
|------|------|
| 要件総数 | N |
| 適合 | N |
| 不適合 | N |
| 要確認 | N |

## 詳細

### {サブタスクタイトル} — 適合
- [x] 要件A: {コードの場所 — `path/to/file.ts:42-60`}

### {サブタスクタイトル} — 不適合
- [ ] 要件B: 未実装 — {何が足りないか}

### {サブタスクタイトル} — 要確認
- [?] 要件A: コードは存在するが動作が不明 — {確認方法の提案}

**凡例**: `[x]` 適合 / `[ ]` 不適合 / `[?]` 要確認（静的解析だけでは判定不能）

## 指摘事項
1. [不適合] {具体的な修正内容}
2. [要確認] {確認が必要な内容}

## 実装ファイル特定の根拠（content にパス記載がなかった場合のみ）

| ファイル | 特定クエリ | ヒット根拠 | 紐付く要件 |
|---|---|---|---|
| {path:line} | `Grep pattern="..."` | {ヒット内容} | {要件 #} |

検索コマンド全文（再現用）: ...
```

## 判定基準

- **適合**: 要件に対応するコードが存在し、ロジックが仕様通り
- **不適合**: 未実装・仕様乖離・制約違反のいずれか
- **要確認**: コードは存在するが静的解析だけでは判断できない

### カスケード規則

- **依存要件が不適合**: ある要件 X が不適合だった場合、X に依存するテスト要件・派生要件は自動的に **要確認 [?]** に格下げする（X が直っていない以上テストの妥当性が判定不能のため）
- **実装ファイルが存在しない**: タスクが「未着手 / WIP」の場合、全要件を **不適合 [ ]**（未実装）として扱い、実装ファイル特定の試行ログを残す
- **判定不能の境界例**: 一部条件のみ満たす実装は **要確認 [?]** にし、何が判定できて何が判定できないかを指摘事項に書き分ける

## 注意事項

- 静的な突合チェックであり、実行時テストは含まない
- 検証レポートはユーザーへの出力として提示し、タスクには書き込まない
