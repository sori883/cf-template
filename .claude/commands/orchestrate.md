# オーケストレートコマンド

複雑なタスクのための順次エージェントワークフロー。

## 前提条件

このコマンドは以下のエージェントが利用可能であることを前提とする:

- `architect`（新規アーキテクチャ設計時）または `planner`（既存コードへの追加・段階的変更時）: 設計・計画
- `tdd-guide`: TDD実装指導
- `code-reviewer`: コード品質レビュー
- `security-reviewer`: セキュリティレビュー
- `explorer`（or `Explore`）: コード探索

`architect` と `planner` の使い分け: ゼロからの新規設計・大規模アーキ変更は `architect`、既存コードベースへの段階的な機能追加・小〜中規模リファクタは `planner`。両方が利用可能な場合はワークフロー（feature/refactor/security）の性質から自動選択する。

### エージェント可用性チェック

ワークフロー起動**前**に以下の順序でチェックを実施する:

1. 必要エージェント（ワークフロー定義に含まれるもののみ。未使用エージェントはチェック対象外）を `~/.claude/agents/` および本リポジトリの `agents/` で確認
2. 結果を可用性レポートとして報告（`[OK] / [MISSING]` の形式）
3. 欠落がある場合、ユーザーに以下の選択肢を提示してから実行に進む:
   - **A**: エージェントをインストールしてから再実行（リポジトリに `agents/<name>.agent.md` がある場合は、そのコピー方法も案内）
   - **B**: inline 代替実行（メイン Claude が該当役割のプロンプト/スキルを直接適用）
   - **C**: 欠落分をスキップして部分実行
   - **D**: キャンセル

ユーザー応答なしで自動実行してはならない（特に security ワークフローではセキュリティ確認のスキップは目的そのものを損なうため、デフォルトで [B] を推奨提示する）。

### Inline 代替の品質扱い

inline 代替（選択肢 B）で実行された役割は、最終レポートで `executed_by: inline-fallback` と明示する。inline 代替の精度は専用エージェントより低下する可能性があるため、最終レポートの「リリース可否」判定では inline 代替分は確証不足として保守的に扱う:

- **リリース可（条件付き）**: inline 代替役割の指摘がすべて軽微で、別途レビューを推奨する場合に使用
- **修正必要**: inline 代替役割で軽微以上の指摘がある場合
- **ブロック中**: inline 代替が品質ゲート役割（security-reviewer / tdd-guide のテストゲート）かつクリティカル領域（認証/PII/決済）の場合

なお、`security` ワークフローでセキュリティ確認を [C] スキップしてはならない（目的そのものを損なうため）。bugfix で品質ゲート役割の tdd-guide が欠落する場合も同様に [B] inline 代替を推奨する。

## 使い方

`/orchestrate [ワークフロータイプ] [タスクの説明]`

## ワークフロータイプ

### feature
機能実装の完全なワークフロー:
```
planner -> tdd-guide -> code-reviewer -> security-reviewer
```

### bugfix
バグ調査と修正のワークフロー:
```
explorer -> tdd-guide -> code-reviewer
```

### refactor
安全なリファクタリングワークフロー:
```
architect -> code-reviewer -> tdd-guide
```

注: refactor ワークフローには security-reviewer を含めない（リファクタは振る舞いを変えない前提のため）。リファクタが認証/PII/権限境界に関わる場合は security ワークフローを別途実行することを最終レポートで推奨する。

### security
セキュリティ重点レビュー:
```
security-reviewer -> code-reviewer -> architect
```

## 実行パターン

ワークフロー内の各エージェントに対して:

1. **エージェントを呼び出し** 前のエージェントからのコンテキストを渡す
2. **出力を収集** 構造化された引き継ぎドキュメントとして
3. **次のエージェントに渡す** チェーン内の次へ
4. **結果を集約** 最終レポートにまとめる

## 引き継ぎドキュメントフォーマット

エージェント間で引き継ぎドキュメントを作成:

```markdown
## HANDOFF: [前のエージェント] -> [次のエージェント]

### コンテキスト
[実行内容の要約]

### 発見事項
[主要な発見や決定]

### 変更されたファイル
[変更されたファイルのリスト]

### 未解決の質問
[次のエージェントへの未解決項目]

### 推奨事項
[提案する次のステップ]
```

**最終エージェントの扱い**: ワークフロー末尾のエージェント（feature なら security-reviewer、refactor なら tdd-guide 等）は HANDOFF を生成せず、その出力は最終レポート（後述）に集約する。最終レポートの該当セクション（`セキュリティステータス` 等）に最終エージェントの発見事項を直接配置する。

## 例: 機能ワークフロー

```
/orchestrate feature "ユーザー認証を追加"
```

実行内容:

1. **Plannerエージェント**
   - 要件を分析
   - 実装計画を作成
   - 依存関係を特定
   - 出力: `HANDOFF: planner -> tdd-guide`

2. **TDD Guideエージェント**
   - plannerの引き継ぎを読み取り
   - テストを先に作成
   - テストをパスするよう実装
   - 出力: `HANDOFF: tdd-guide -> code-reviewer`

3. **Code Reviewerエージェント**
   - 実装をレビュー
   - 問題をチェック
   - 改善を提案
   - 出力: `HANDOFF: code-reviewer -> security-reviewer`

4. **Security Reviewerエージェント**
   - セキュリティ監査
   - 脆弱性チェック
   - 最終承認
   - 出力: 最終レポート

## 最終レポートフォーマット

```
オーケストレーションレポート
====================
ワークフロー: feature
タスク: ユーザー認証を追加
エージェント: planner -> tdd-guide -> code-reviewer -> security-reviewer

サマリー
-------
[1段落の要約]

エージェント出力
-------------
Planner: [要約]
TDD Guide: [要約]
Code Reviewer: [要約]
Security Reviewer: [要約]

変更されたファイル
-------------
[変更された全ファイルのリスト]

テスト結果
------------
[テストのパス/フェイルサマリー]

セキュリティステータス
---------------
[セキュリティの発見事項]

推奨
--------------
[リリース可 / 修正必要 / ブロック中]
```

## 並列実行

独立したチェックの場合、エージェントを並列実行:

```markdown
### 並列フェーズ
同時に実行:
- code-reviewer（品質）
- security-reviewer（セキュリティ）
- architect（設計）

### 結果のマージ
出力を単一のレポートに統合
```

## 引数

$ARGUMENTS:
- `feature <説明>` - 機能実装の完全なワークフロー
- `bugfix <説明>` - バグ修正ワークフロー
- `refactor <説明>` - リファクタリングワークフロー
- `security <説明>` - セキュリティレビューワークフロー
- `custom <エージェント> <説明>` - カスタムエージェントシーケンス

## カスタムワークフローの例

```
/orchestrate custom "architect,tdd-guide,code-reviewer" "キャッシュレイヤーを再設計"
```

## ヒント

1. **複雑な機能はplannerから開始** する
2. **マージ前には常にcode-reviewerを含める**
3. **認証/決済/PIIにはsecurity-reviewerを使用**
4. **引き継ぎは簡潔に** - 次のエージェントが必要とする情報に焦点を当てる
5. **必要に応じてエージェント間で検証を実行**
