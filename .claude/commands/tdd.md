# TDDコマンド

このコマンドは**tdd-guide**エージェントを呼び出し、テスト駆動開発の方法論を強制する。本コマンドは言語非依存であり、検出されたプロジェクト言語に応じて適切なテストランナー・カバレッジツールを使用する。

## このコマンドの機能

1. **型/インターフェースの定義** - データ構造とAPIを最初に定義
2. **テストを先に生成** - 失敗するテストを書く（RED）
3. **最小限のコードで実装** - パスするのに十分なコードのみ書く（GREEN）
4. **リファクタリング** - テストをグリーンに保ちながらコードを改善（REFACTOR）
5. **カバレッジを検証** - 80%以上のテストカバレッジを確保

## 言語/ツール検出

実行前にプロジェクトのマニフェストから言語を検出し、以下のツールを選択する:

| 言語 | テストランナー | カバレッジ |
|------|--------------|----------|
| Rust | `cargo test` | `cargo tarpaulin` または `cargo llvm-cov` |
| TypeScript/JavaScript | `vitest` / `jest` / `node --test` | 同ランナーの `--coverage` |
| Python | `pytest` | `pytest --cov` または `coverage` |
| Go | `go test ./...` | `go test -cover ./...` |
| Java | `mvn test` / `gradle test` | `jacoco` |
| その他 | プロジェクトの README/CI 設定から推論 | 同左 |

## 使用タイミング

以下の場合に `/tdd` を使用:
- 新機能の実装
- 新しい関数/モジュールの追加
- バグ修正（まずバグを再現するテストを書く）
- 既存コードのリファクタリング
- クリティカルなビジネスロジックの構築

## 動作の仕組み

tdd-guideエージェントが以下を実行:

1. **入出力の型/インターフェースを定義**
2. **失敗するテストを書く** （コードがまだ存在しないため）
3. **テストを実行** し正しい理由で失敗することを確認
4. **最小限の実装を書く** テストをパスさせるため
5. **テストを実行** しパスすることを確認
6. **コードをリファクタリング** テストをグリーンに保ちながら
7. **カバレッジを確認** 80%未満ならテストを追加

## TDDサイクル

```
RED → GREEN → REFACTOR → REPEAT

RED:      失敗するテストを書く
GREEN:    パスするための最小限のコードを書く
REFACTOR: テストをパスさせたままコードを改善
REPEAT:   次の機能/シナリオへ
```

## 使用例（言語非依存）

ユーザー: `/tdd タスクのフィルタリング関数が必要`

エージェント（tdd-guide）の流れ:

### ステップ1: 型/シグネチャ定義（SCAFFOLD）

入出力を最初に確定させる。例（疑似コード）:

```
struct TaskFilter {
  project: Optional<String>
  label: Optional<String>
  priority: Optional<Priority>
  include_done: bool
}

function filter_tasks(tasks: List<Task>, filter: TaskFilter) -> List<Task>
  // 未実装（テストを先に書くため）
```

実装言語に応じた具体的な構文に展開する（Rust の `struct` + `pub fn`、TypeScript の `interface` + `function`、Python の `dataclass` + `def` 等）。

### ステップ2: 失敗するテストを書く（RED）

最低3つのケースを書く:

1. **正常系**: フィルタが期待通り動く（例: project="web" を指定 → web の active タスクのみ返る）
2. **フラグの効果**: オプションフラグの効果を検証（例: `include_done=true` で完了タスクも含む）
3. **デフォルト挙動**: 空フィルタのデフォルト動作（例: 完了は除外して返す）

各テストはサンプルデータを用意し、入力 → 期待出力 を `assert` する。

### ステップ3: テスト実行 - 失敗を確認

検出されたテストランナーで実行し、3件全てが失敗することを確認する。失敗理由が「未実装」であることを確認する（コンパイルエラーや構文エラーではない）。

### ステップ4: 最小限のコードで実装（GREEN）

テストをパスさせる最小限のコードを書く。早すぎる最適化はしない。例（擬似コード）:

```
function filter_tasks(tasks, filter):
  result = []
  for t in tasks:
    if not filter.include_done and t.status == Done:
      continue
    if filter.project is set and t.project != filter.project:
      continue
    result.append(t)
  return result
```

### ステップ5: テスト実行 - パスを確認

全テストがパスすることを確認。1件でも失敗したら GREEN 完了ではない。

### ステップ6: リファクタリング（IMPROVE）

テストがグリーンのまま、可読性・パフォーマンス・抽象度を改善。各リファクタリング後にテストを再実行。

### ステップ7: カバレッジの確認

検出された言語のカバレッジツールで対象モジュールのカバレッジを測定し、80%未満ならテストを追加する。

## TDDベストプラクティス

**推奨:**
- 実装の前にまずテストを書く
- 実装前にテストを実行し失敗を確認
- テストをパスさせるための最小限のコードを書く
- テストがグリーンになってからリファクタリング
- エッジケースとエラーシナリオを追加
- 80%以上のカバレッジを目標（クリティカルなコードは100%）

**非推奨:**
- テスト前に実装を書く
- 各変更後のテスト実行をスキップ
- 一度に大量のコードを書く
- 失敗するテストを無視
- 実装の詳細をテスト（振る舞いをテストすること）

## 含めるべきテストタイプ

**ユニットテスト**:
- 個別の関数とユーティリティ
- 構造体/クラスのメソッド
- エラーケース（例外/Result::Err/エラーリターンパス）
- 境界値

**統合テスト**:
- サービス層の連携テスト
- リポジトリ + サービスの結合
- ファイル/DB I/O

**E2Eテスト**（CLI/API の場合）:
- コマンド/エンドポイントの実行と出力検証
- 出力フォーマットのパース
- エラーメッセージの検証

## カバレッジ要件

- **最低80%** 全コードに対して
- **100%必須** 以下のコードに対して:
  - データの永続化ロジック
  - 入力バリデーション/サニタイズ
  - エラーハンドリングパス
  - コアビジネスロジック

## 重要な注意事項

**MANDATORY**: テストは実装の前に書くこと。TDDサイクルは:

1. **RED** - 失敗するテストを書く
2. **GREEN** - パスするよう実装
3. **REFACTOR** - コードを改善

REDフェーズを絶対にスキップしないこと。テスト前にコードを書かないこと。
