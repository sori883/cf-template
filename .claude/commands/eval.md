# Evalコマンド

Eval駆動開発ワークフローを管理する。Eval は `.claude/evals/` 配下に Markdown ファイルとして保存される。実行ログは `.claude/evals/<機能名>.log` に追記される。

## 使い方

`/eval [define|check|report|list|clean] [機能名]`

## 用語定義

- **試行（trial）**: `/eval check 機能名` の1回の実行 = 1試行とカウント
- **pass@k**: 直近 k 試行のうち**少なくとも1回**成功すれば PASS
- **pass^k**: 直近 k 試行で**すべて**成功すれば PASS
- **ログエントリの単位**: 1試行 = 1ブロック。ヘッダ行は `===== TRIAL <YYYY-MM-DD HH:MM> =====` で開始
- **ログ行フォーマット**: `<eval-id> | <PASS|FAIL> | <note>`（パイプ区切り、note は空可）

## Evalの定義

`/eval define 機能名`

新しいEval定義を作成:

1. `.claude/evals/` ディレクトリが存在しなければ作成（`mkdir -p .claude/evals`）
2. `.claude/evals/<機能名>.md` にテンプレートを作成（`機能名` は引数の値で置換、`$(date)` は実行時の日付に置換）:

```markdown
## EVAL: <機能名>
作成日: <YYYY-MM-DD>

### 機能Eval
- [ ] [機能1の説明]
- [ ] [機能2の説明]

### リグレッションEval
- [ ] [既存動作1が引き続き動作する]
- [ ] [既存動作2が引き続き動作する]

### 成功基準
- 機能Evalのpass@3 > 90%
- リグレッションEvalのpass^3 = 100%
```

3. ユーザーに具体的な基準の入力を促す（チェックボックスの内容を埋めるよう案内）

## Evalのチェック

`/eval check 機能名`

機能のEvalを実行（1回の実行 = 1試行）:

1. `.claude/evals/<機能名>.md` からEval定義を読み取り
2. 各機能Evalについて以下を実行:
   - 基準の検証を試行（検証手段は基準の記述から推論: 自動テストコマンド/CLI実行/LLM判定/ファイル存在チェック等）
   - PASS/FAILを判定
3. 各リグレッションEvalについて:
   - 関連する既存テストを実行（プロジェクトの検出されたテストランナーを使用）
   - 結果をベースラインと比較
   - PASS/FAIL を判定
4. **ログ追記**: `.claude/evals/<機能名>.log` に **append モード**で1試行ブロックを追記:

```
===== TRIAL 2026-05-03 10:15 =====
eval-1 | PASS |
eval-2 | FAIL | エラー: timeout
test-1 | PASS |
test-2 | PASS |
```

5. 現在のステータスを報告:

```
EVAL CHECK: <機能名>
========================
機能: X/Y パス
リグレッション: X/Y パス
ステータス: 未開始 / 進行中 / 準備完了
```

### ステータス判定ロジック

`.claude/evals/<機能名>.log` の試行履歴を読み、以下のルールで判定:

- **未開始**: ログファイルが存在しない、または試行0回
- **準備完了**: 直近3試行で「機能Eval pass@3 > 90%」かつ「リグレッションEval pass^3 = 100%」を満たす
- **進行中**: 上記以外（試行はあるが成功基準未達）

## Evalレポート

`/eval report 機能名`

`.claude/evals/<機能名>.log` の全試行ブロックを集計し、包括的なレポートを生成:

集計手順:
1. ログから全試行ブロックを読み取り、各 eval-id の PASS/FAIL 履歴を構築
2. 機能Eval: 直近3試行で pass@3 を計算（少なくとも1回 PASS で OK）
3. リグレッションEval: 直近3試行で pass^3 を計算（3回連続 PASS で OK）
4. 各 Eval ごとに、最初に PASS した試行番号（pass@N）を記録
5. 全体の最終推奨を以下のルールで決定:
   - **リリース可**: 機能 pass@3 > 90% かつ リグレッション pass^3 = 100%
   - **修正必要**: 上記の片方のみ満たす
   - **ブロック中**: 両方満たさず、または試行不足

レポートフォーマット:

```
EVAL REPORT: 機能名
=========================
生成日: $(date)

機能EVAL
----------------
[eval-1]: PASS (pass@1)
[eval-2]: PASS (pass@2) - リトライが必要だった
[eval-3]: FAIL - 注記を参照

リグレッションEVAL
----------------
[test-1]: PASS
[test-2]: PASS
[test-3]: PASS

メトリクス
-------
機能 pass@1: 67%
機能 pass@3: 100%
リグレッション pass^3: 100%

注記
-----
[問題、エッジケース、または所見]

推奨
--------------
[リリース可 / 修正必要 / ブロック中]
```

## Eval一覧

`/eval list`

全Eval定義を表示:

```
EVAL定義一覧
================
feature-auth      [3/5パス] 進行中
feature-search    [5/5パス] 準備完了
feature-export    [0/4パス] 未開始
```

## Evalログのクリーン

`/eval clean`

古い Eval ログを削除する:

1. `.claude/evals/*.log` の各ファイルを処理対象とする
2. 各ファイル内の試行ブロック（`===== TRIAL ... =====` で始まる）を読み取り、タイムスタンプ降順でソート
3. 最新10ブロックを保持し、それより古いブロックを削除
4. 削除対象が0件のファイルはスキップ
5. 各ファイルごとに削除件数を表示し、合計件数を最後にサマリ:

```
EVAL CLEAN
==========
feature-auth.log:    削除: 5件 / 保持: 10件
feature-search.log:  削除: 0件 / 保持: 8件
feature-export.log:  削除: 12件 / 保持: 10件

合計: 17件削除 / 28件保持
```

## 引数

$ARGUMENTS:
- `define <名前>` - 新しいEval定義を作成
- `check <名前>` - Evalを実行しチェック（1回 = 1試行）
- `report <名前>` - 完全なレポートを生成
- `list` - 全Evalを表示
- `clean` - 古い Eval ログ試行ブロックを削除（各ファイルで最新10ブロック保持）
