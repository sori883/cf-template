# /learn - 再利用可能なパターンの抽出

現在のセッションを分析し、スキルとして保存する価値のあるパターンを抽出する。

## トリガー

ユーザーが `/learn` を明示的に呼び出した時点で実行する。以下のいずれかが直前のセッションで発生していれば抽出対象がある可能性が高い:

- ドキュメントだけからは到達できなかった原因究明（ログ深堀り、二分探索デバッグ、実環境再現）
- 30分以上かかったエラー解決
- ライブラリ/フレームワークの非自明な挙動の発見
- 同種タスクで再利用できそうな手順の確立
- 通常の手順では失敗するケースに対するワークアラウンド

以下に該当する場合は抽出をスキップ:
- タイプミスや単純な構文エラーの修正
- ドキュメントに直接記載されている手順
- そのプロジェクトのみに特化した内容で再利用性がない

## 抽出対象

以下を探す:

1. **エラー解決パターン**
   - どのようなエラーが発生したか？
   - 根本原因は何だったか？
   - 何が修正したか？
   - 類似のエラーに再利用可能か？

2. **デバッグテクニック**
   - 非自明なデバッグ手順
   - 有効だったツールの組み合わせ
   - 診断パターン

3. **ワークアラウンド**
   - ライブラリの癖
   - APIの制限
   - バージョン固有の修正

4. **プロジェクト固有のパターン**
   - 発見したコードベースの慣習
   - 行われたアーキテクチャ上の決定
   - 統合パターン

## 出力フォーマット

`~/.claude/skills/learned/[パターン名].md` にスキルファイルを作成:

```markdown
# [説明的なパターン名]

**抽出日:** [日付]
**コンテキスト:** [適用場面の簡単な説明]

## 問題
[このパターンが解決する問題 - 具体的に]

## 解決策
[パターン/テクニック/ワークアラウンド]

## 例
[該当する場合のコード例]

## 使用タイミング
[トリガー条件 - このスキルを発動させるべき状況]
```

## プロセス

1. セッションを確認し、抽出可能なパターンを探す
2. 最も価値のある/再利用可能なインサイトを特定
3. スキルファイルのドラフトを作成
4. 保存前にユーザーの確認を得る
5. `~/.claude/skills/learned/` に保存

## 注意事項

- 些細な修正（タイプミス、単純な構文エラー）は抽出しない
- 一回限りの問題（特定の API 一時障害など）は抽出しない
- 将来のセッションで時間を節約できるパターンに焦点を当てる
- スキルは焦点を絞る - 1スキルにつき1パターン
- 言語/フレームワーク固有のパターンを抽出する場合、適用条件（例: 「TypeScript + Node.js >= 18 で発生」）を明記する
