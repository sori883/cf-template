# リファクタリング・クリーン

テスト検証によるデッドコードの安全な特定と削除:

## 言語/ツール検出

プロジェクトのマニフェストに応じて以下のツールを使用する:

| 言語 | デッドコード検出ツール |
|------|---------------------|
| TypeScript/JavaScript | `knip`（未使用ファイル/エクスポート）、`depcheck`（未使用依存）、`ts-prune`（未使用エクスポート） |
| Rust | `cargo-udeps`（未使用依存）、`cargo machete`、`cargo +nightly rustc -- -W dead_code` |
| Python | `vulture`（未使用コード）、`deptry`（未使用依存） |
| Go | `staticcheck`（U1000など）、`go mod tidy` |
| Java/Kotlin | IDE inspection、`mvn dependency:analyze` |
| その他 | プロジェクトのlinterのdead-code検出ルール |

利用可能なツールがプロジェクトに未導入の場合、ユーザーに導入の可否を確認してから進める。

## 手順

1. 検出されたデッドコード分析ツールを実行
   - **モノレポの場合**: ルートに workspace 定義がある場合（`pnpm-workspace.yaml`、`Cargo.toml [workspace]`、`go.work` 等）、ツール側のモノレポ対応（例: `pnpm exec knip`、`cargo machete --workspace`）でルート一括実行する。各 workspace で個別実行が必要なツールは workspace 毎に並列実行。

2. 包括的なレポートを `.reports/dead-code-analysis.md` に生成（`.reports/` ディレクトリは存在しなければ作成する）

3. 検出結果を重要度別に分類:
   - **安全**: テストファイル内、ローカルのユーティリティ関数、未使用の private 関数（先頭 `_` 等）
   - **注意**: API ルート/エンドポイント、コンポーネント、public 関数（外部参照の可能性）
   - **危険**: 設定ファイル、メインエントリーポイント、エクスポートされた型/インターフェース、`__init__.py` の re-export、entry_points

4. 安全な削除のみを提案（注意・危険レベルは確認後）

### テストランナーの検出

- **package.json `scripts.test`** が存在すれば優先使用（`npm test` / `pnpm test` 等）
- マニフェストから推論できない場合は CI 設定（`.github/workflows/`、`Makefile`）からテストコマンドを抽出
- モノレポでは workspace 全体をカバーするコマンド（`pnpm -r test`、`cargo test --workspace`、`go test ./...`、`pytest`）を採用

### 削除の最小単位（粒度）

「1項目ずつ」とは以下の単位を指す:

- **未使用ファイル削除**: 1ファイル = 1項目
- **未使用 export 削除**: 1 (file, exported symbol) ペア = 1項目
- **未使用依存削除**: 1依存 = 1項目（マニフェストから1エントリ削除）
- 同一ファイル内の複数 export を一括削除する場合でも、テスト実行は各 export ごとに行う

### 削除サイクル

5. **削除粒度: 1項目ずつ**。各削除の前に以下のサイクルを実行:
   - 全テストスイートを実行（検出されたテストランナー）
   - テストがパスすることを確認
   - 変更を適用（コード削除、または `package.json`/`Cargo.toml`/`pyproject.toml` 等からの依存削除）
   - テストを再実行
   - **テストが失敗した場合はロールバック**:
     - コード変更: 未コミット状態なら `git checkout -- <files>` で取消
     - 依存削除: マニフェストを元に戻し、依存復元コマンド（`pnpm install`、`cargo build`、`pip install -e .` 等）を再実行
     - コミット済みなら `git revert` を作成
   - 失敗したケースは `.reports/dead-code-analysis.md` に「ロールバック理由」と共に記録し、安全な削除リストから除外する

6. クリーンアップ項目のサマリーを表示（削除成功件数、ロールバック件数、保留件数を区別）

### テスト不在時のフォールバック

- テストランナーが検出できない、またはテストが0件の場合:
  - 「テストカバレッジが不十分」と警告
  - ユーザーに以下の選択肢を提示:
    - **A**: 分析レポートのみ生成し、削除は実施しない
    - **B**: 削除前に最低限のテストを追加してから再実行
    - **C**: リスクを承知の上で手動レビュー用にレポート提供（**自動削除はしない**）
  - ユーザーの明示的承認なく自動削除を実行してはならない

### ツール未導入時のフォールバック

- 検出ツールがプロジェクトに未導入の場合、ユーザーに導入の可否を確認
- ユーザーが導入を拒否した場合、利用可能な代替ツール（言語標準のlinter等）で部分実行するか、分析を中止して状態を報告する

テストを実行せずにコードを削除してはならない。
